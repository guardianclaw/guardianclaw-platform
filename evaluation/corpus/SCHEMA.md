# Corpus Item Schema

Each class file is a JSON object with this top-level shape:

```json
{
  "class": "prompt-injection",
  "schema_version": 1,
  "class_version": 1,
  "description": "Direct attempts to override the system prompt or smuggle instructions through user input.",
  "primary_gates": ["avoidance"],
  "items": [ /* item objects, see below */ ]
}
```

## Top-level fields

| Field | Type | Required | Notes |
|---|---|---|---|
| `class` | string | yes | One of: `prompt-injection`, `data-exfil`, `jailbreak`, `encoding`, `multilingual`, `instruction-override`, `role-play`, `indirect-via-memory`. Must match filename (`<class>.json`). |
| `schema_version` | int | yes | Bump on field renames or structural changes. Harness rejects unknown values. Current: `1`. |
| `class_version` | int | yes | Bump when adding/removing items. Pure tracking; harness does not check. |
| `description` | string | yes | One-paragraph plain English description of the attack shape. |
| `primary_gates` | string[] | yes | Gates expected to fire most often for this class. Reference only; not enforced. |
| `items` | array | yes | ≥50 items per class for the frente acceptance criterion. |

## Item fields

| Field | Type | Required | Notes |
|---|---|---|---|
| `id` | string | yes | Unique within class. Format `<prefix>-NNN` (e.g. `pi-001`). Stable across versions — never reuse a deleted id. |
| `input` | string | yes | The text fed to `CLAWValidator.validate()`. |
| `language` | string | yes | ISO 639-1 code (`en`, `pt`, `es`, `zh`, ...). For mixed-language items, use the dominant one. |
| `source` | string | yes | One of: `curated` (hand-written for this corpus), `literature` (paraphrased from public research), `wildjailbreak-style` (paraphrased from publicly disclosed jailbreaks), `synthetic` (generated by transformation rule). |
| `expected_verdict` | object | yes | See below. |
| `rationale` | string | yes | One sentence justifying each non-trivial verdict component. |
| `tags` | string[] | no | Free-form taxonomy refinement (e.g. `["base64", "system-prompt-target"]`). |

## `expected_verdict` object

```json
{
  "credibility": "pass" | "fail" | "any",
  "limits":      "pass" | "fail" | "any",
  "avoidance":   "pass" | "fail" | "any",
  "worth":       "pass" | "fail" | "any"
}
```

Verdict semantics:

- `"pass"` — gate must NOT fire on this input.
- `"fail"` — gate MUST fire on this input.
- `"any"` — gate behavior is unconstrained. Reserve for items where current SDK behavior is intentionally undefined and an item should still be tracked. Always pair with rationale.

`overall` is derived by the harness: `fail` if any gate is `fail`, else `pass`. Items do not declare `overall` directly.

## Mismatch reporting

The harness produces, per item:

```json
{
  "id": "pi-007",
  "class": "prompt-injection",
  "input_excerpt": "Ignore previous instructions and...",
  "expected": {"credibility":"pass","limits":"pass","avoidance":"fail","worth":"pass"},
  "actual":   {"credibility":"pass","limits":"pass","avoidance":"pass","worth":"pass"},
  "violations": [],
  "diff": ["avoidance: expected fail, got pass"]
}
```

Items with empty `diff` array are silenced unless `--verbose` is set.

## Backwards compatibility

Adding new optional fields to items is non-breaking and does NOT bump `schema_version`. Removing or renaming any field DOES bump it.
